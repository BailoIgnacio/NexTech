================================================================================
 4. COMO CORRER EL PROYECTO
================================================================================

Requisitos: Python 3.10 o superior instalado.

Pasos:
  1. Abrir terminal en la carpeta backend/:
       cd backend

  2. Instalar dependencias (solo la primera vez):
       pip install -r requirements.txt

  3. Iniciar el servidor:
       python -m uvicorn main:app --reload --port 3000

  4. Abrir en el navegador:
       http://localhost:3000/           <- Tienda (frontend)
       http://localhost:3000/admin/     <- Panel de administracion
       http://localhost:3000/docs       <- Swagger UI (documentacion interactiva de la API)
       http://localhost:3000/redoc      <- ReDoc (alternativa a Swagger)

Importante: el servidor debe correr desde dentro de la carpeta backend/ porque
storage.py construye rutas relativas al __file__ para encontrar backend/data/.

 6. API REST - ENDPOINTS DETALLADOS
================================================================================

URL base: http://localhost:3000/api
Formato: JSON
Autenticacion: NINGUNA (los endpoints de escritura NO requieren token en la
implementacion actual - esto es un bug intencional para testing)


--- PRODUCTOS ---

GET /api/products
  Descripcion: Lista todos los productos con soporte de filtros y paginacion.
  Query params opcionales:
    categoria   : string exacto ("Laptops" | "Celulares" | "Auriculares" | "Accesorios")
    color       : string exacto ("Negro" | "Blanco" | "Gris" | "Plateado")
    destacado   : "true" | "false"
    novedad     : "true" | "false"
    search      : string (busca en nombre, descripcion y marca - CASE SENSITIVE, bug intencional)
    precio_min  : float
    precio_max  : float
    sort        : "nombre" | "precio" | "fecha"
    order       : "asc" | "desc" (default: "asc")
    page        : int (default: 1)
    limit       : int (default: 12)
  Respuesta 200:
    {
      "data": [...],      // lista de productos de la pagina actual
      "total": 16,        // total de productos tras aplicar filtros
      "page": 1,
      "totalPages": 2,
      "limit": 12
    }

GET /api/products/{id}
  Descripcion: Obtiene un producto por su UUID.
  Respuesta 200: objeto producto completo
  Respuesta 404: {"detail": "Producto no encontrado"}

POST /api/products
  Descripcion: Crea un producto nuevo.
  Body requerido (JSON):
    nombre  : string (requerido, no puede estar vacio)
    precio  : float  (requerido, debe ser > 0)
    + todos los demas campos de ProductCreate son opcionales con defaults
  Respuesta 201: objeto producto creado (con id y fechas generados)
  Respuesta 400: {"detail": "El campo nombre es requerido"} o {"detail": "El campo precio debe ser mayor a 0"}
  Respuesta 422: error de validacion de Pydantic (campos faltantes o tipos incorrectos)
  NOTA: No requiere autenticacion (bug intencional)

PUT /api/products/{id}
  Descripcion: Reemplaza un producto completo.
  Body: mismo schema que POST /api/products (ProductCreate completo)
  Aplica las mismas validaciones de nombre y precio.
  Respuesta 200: objeto producto actualizado
  Respuesta 404: si el id no existe

PATCH /api/products/{id}
  Descripcion: Actualizacion parcial de un producto. Solo envia los campos a cambiar.
  Body: cualquier combinacion de campos de ProductPatch (todos opcionales)
  IMPORTANTE - BUG INTENCIONAL: PATCH NO valida que precio sea > 0.
    Permite enviar precio=-100 y lo acepta (a diferencia de POST y PUT).
  Respuesta 200: objeto producto actualizado
  Respuesta 404: si el id no existe

DELETE /api/products/{id}
  Descripcion: Elimina un producto.
  Respuesta 204: sin cuerpo (exito)
  Respuesta 404: {"detail": "Producto no encontrado"}
  NOTA: No requiere autenticacion (bug intencional)


--- AUTENTICACION ---

POST /api/auth/register
  Descripcion: Registra un nuevo usuario.
  Body (JSON):
    nombre   : string (requerido, no puede estar vacio)
    email    : string (requerido, no puede estar vacio)
    password : string (requerido)
  Respuesta 201:
    {
      "id": "...",
      "nombre": "...",
      "email": "...",
      "rol": "usuario"
    }
  Respuesta 400: si nombre, email o password estan vacios
  IMPORTANTE - BUG INTENCIONAL: NO verifica si el email ya existe.
    Permite registrar dos usuarios con el mismo email.
  IMPORTANTE - BUG SEGURIDAD: La password se guarda en texto plano en users.json.

POST /api/auth/login
  Descripcion: Autentica un usuario existente.
  Body (JSON):
    email    : string
    password : string
  Respuesta 200:
    {
      "mensaje": "Login exitoso",
      "usuario": {
        "id": "...",
        "nombre": "...",
        "email": "...",
        "rol": "admin" | "operador" | "usuario"
      }
    }
  Respuesta 401: {"detail": "Credenciales incorrectas"}
  NOTA: La respuesta NO devuelve ningun token JWT ni session token.
    El frontend no implementa autenticacion real (no hay sesion persistente entre
    paginas). El login solo valida credenciales y devuelve datos del usuario.


--- ESTADISTICAS ---

GET /api/stats
  Descripcion: Devuelve estadisticas del catalogo de productos.
  Respuesta 200:
    {
      "total_productos": 16,
      "total_destacados": 4,
      "total_novedades": 6,
      "total_sin_stock": 2
    }
  IMPORTANTE - BUG/INCONSISTENCIA: total_sin_stock se calcula contando productos
    donde stock == 0. Sin embargo, existe el campo con_stock que es un boolean
    separado que puede estar desincronizado. El endpoint ignora con_stock al
    calcular sin_stock y usa unicamente el campo numerico stock.

================================================================================
 9. BACKEND - LOGICA INTERNA
================================================================================

--- backend/main.py ---
  - Crea la instancia FastAPI con titulo "NexTech API" version "1.0.0"
  - Middleware CORS: permite todos los origenes (allow_origins=["*"])
  - Registra los 3 routers con prefijo /api
  - Monta el frontend estatico en la raiz "/" con html=True (sirve index.html por defecto)
  - Logging: escribe en backend/logs/app.log Y en consola simultaneamente

--- backend/storage.py ---
  - read_json(filename) : lee backend/data/{filename} y retorna el dict Python
  - write_json(filename, data) : serializa y escribe con indent=2, ensure_ascii=False
  - No tiene cache ni concurrencia. Cada request lee el archivo completo.

--- backend/routes/products.py ---
  - Pydantic models: ProductCreate (campos requeridos: nombre, precio) y ProductPatch (todo opcional)
  - Todos los filtros de GET /products se aplican en Python sobre la lista leida del JSON
  - IDs generados con uuid.uuid4()
  - Fechas generadas con datetime.utcnow().isoformat()

--- backend/routes/auth.py ---
  - Pydantic models: RegisterRequest, LoginRequest
  - No usa JWT, no tiene middleware de autorizacion
  - Login: busqueda por coincidencia exacta de email Y password en texto plano
  - La respuesta de login NO incluye el campo password

--- backend/routes/stats.py ---
  - Lee products.json, calcula conteos en memoria y los devuelve


================================================================================
 10. BUGS CONOCIDOS E INTENCIONALES
================================================================================

Estos bugs son INTENCIONALES para que los testers los encuentren como parte del TP.
El Developer debera corregirlos cuando los Testers los reporten como defectos.

BUG-01 (Auth - Registro de email duplicado)
  Archivo: backend/routes/auth.py, funcion register()
  El comentario en el codigo dice: "Registra el usuario sin verificar si el email ya existe"
  Efecto: POST /api/auth/register con email ya existente retorna 201 en lugar de 409.
  Resultado esperado correcto: deberia retornar 409 Conflict con mensaje de error.

BUG-02 (Productos - Busqueda case-sensitive)
  Archivo: backend/routes/products.py, funcion get_products()
  El comentario dice: "La busqueda es sensible a mayusculas/minusculas"
  Efecto: GET /api/products?search=iphone no encuentra "iPhone 15 Pro".
          GET /api/products?search=iPhone si lo encuentra.
  Resultado esperado correcto: la busqueda deberia ser case-insensitive.

BUG-03 (Productos - PATCH no valida precio negativo)
  Archivo: backend/routes/products.py, funcion partial_update_product()
  El comentario dice: "sin validar precio en PATCH"
  Efecto: PATCH /api/products/{id} con {"precio": -100} retorna 200 y guarda el precio negativo.
          POST y PUT si validan que precio > 0 y retornan 400.
  Resultado esperado correcto: PATCH tambien deberia rechazar precios <= 0.

BUG-04 (Stats - Inconsistencia con_stock vs stock)
  Archivo: backend/routes/stats.py
  total_sin_stock se calcula con: sum(1 for p in products if p.get("stock", 0) == 0)
  Pero existe el campo booleano con_stock que puede estar desincronizado con el stock numerico.
  Efecto: un producto con stock=5 y con_stock=False no seria contado como sin stock.
          Un producto con stock=0 y con_stock=True si seria contado como sin stock.
  El campo con_stock es un campo manual que no se actualiza automaticamente.

BUG-05 (Datos - Marca incorrecta en Google Pixel 8)
  Archivo: backend/data/products.json
  El Google Pixel 8 tiene marca="Xiaomi" en lugar de "Google".
  Efecto: buscar ?search=Google no encuentra el Pixel 8. Buscar ?marca=Xiaomi lo incluye.

BUG-06 (Autorizacion ausente en endpoints de escritura)
  Archivos: backend/routes/products.py
  Los endpoints POST, PUT, PATCH y DELETE de /api/products no requieren autenticacion.
  Cualquier persona puede crear, modificar o eliminar productos sin estar logueada.
  El panel admin tampoco tiene proteccion de login (accesible en /admin/ directamente).
  Resultado esperado correcto: deberian requerir un token y que el rol sea "admin".

BUG-07 (Seguridad - Passwords en texto plano)
  Archivo: backend/data/users.json y backend/routes/auth.py
  Las passwords se almacenan y comparan en texto plano, sin hash.
  Resultado esperado correcto: deberian hashearse con bcrypt u similar.

BUG-08 (Auth - Sin validacion de longitud de password)
  Archivo: backend/routes/auth.py, funcion register()
  El RNF-02 establece que las contrasenas deben tener minimo 8 caracteres.
  Pero la API acepta passwords de cualquier longitud (incluso 1 caracter) sin error.
  Resultado esperado correcto: deberia retornar 400 si la password tiene menos de 8 caracteres.

 14. CONFIGURACION DE LOGGING

El sistema registra logs en:
  - Consola (stdout)
  - backend/logs/app.log

Formato: "TIMESTAMP [LEVEL] logger_name - mensaje"
Ejemplo: "2026-05-11 10:30:00,123 [INFO] nextech.products - GET /products - total=16 page=1 limit=12"

Loggers:
  - nextech           (main.py)
  - nextech.products  (routes/products.py)

  - nextech.stats     (routes/stats.py)

  - Inicio de la aplicacion
  - Cada GET /products (con total, page, limit)
  - Cada producto creado (con id y nombre)
  - Cada producto actualizado (PUT y PATCH)
  - Cada producto eliminado
  - Login exitoso (con email y rol)
  - Login fallido (con email)
  - Producto no encontrado (GET por id)
  - GET /stats (con totales)


================================================================================
 15. GUIA RAPIDA PARA TRABAJAR CON EL PROYECTO
================================================================================

Para agregar un endpoint nuevo:
  1. Crear o editar un archivo en backend/routes/
  2. Definir el router con APIRouter()
  3. Si es archivo nuevo, registrarlo en backend/main.py: app.include_router(nuevo.router, prefix="/api")
  4. La persistencia se hace via storage.read_json() y storage.write_json()
  5. Para un nuevo tipo de datos, crear backend/data/nuevo_tipo.json con estructura inicial

Para corregir un bug:
  - Los bugs de validacion estan en backend/routes/products.py y backend/routes/auth.py
  - Despues de corregir, correr pytest tests/ -v para verificar que los tests siguen pasando
  - Puede ser necesario agregar nuevos tests que cubran el caso corregido

Para modificar el frontend:
  - Los archivos JS usan ES6 modules (import/export) sin bundler
  - Los cambios en archivos estaticos se reflejan inmediatamente (no hay build)
  - El frontend consume la API via js/api.js (no hacer fetch directamente)

Para ver la documentacion de la API:
  - Con el servidor corriendo: http://localhost:3000/docs (Swagger UI interactivo)
  - Permite probar endpoints directamente desde el navegador
